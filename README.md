# Four Horses Club — лендинг
Досупен публично: https://four-horses-club-landing.pages.dev/.

Этот README — пошаговая инструкция как сверстать такой лендинг от пустой структуры проекта до адаптивной страницы.

![Первый экран](./assets/docs/first-screen.png)

## 1. Подготовить структуру проекта

Создаем в корне проекта предсказуемую структуру. Задача такой структуры — не смешивать разметку, стили, скрипты и ассеты. 
Все CSS-файлы собираются через `css/styles.css`, а `index.html` подключает только итоговый файл стилей.

```text
four-horses-club/
├─ index.html
├─ README.md
├─ assets/
│  ├─ images/
│  ├─ icons/
│  └─ fonts/
├─ css/
│  ├─ reset.css
│  ├─ variables.css
│  ├─ base.css
│  ├─ layout.css
│  ├─ fonts.css
│  ├─ components.css
│  └─ styles.css
└─ js/
   ├─ participants-carousel.js
   └─ plan-slider.js
```

- `reset.css` — сброс браузерных стилей;
- `variables.css` — цвета, состояния кнопок, общие токены;
- `layout.css` — контейнер и общая сетка страницы;
- `fonts.css` — `@font-face`;
- `base.css` — базовые правила страницы и утилиты;
- `components.css` — конкретные блоки лендинга.

Так проще поддерживать проект: общие вещи остаются общими, а стили секций живут рядом друг с другом в компонентном файле.

## 2. Подготовить ассеты

Экспортируем изображения из Figma в `assets/images/`, слишком тяжелые файлы сжимаем. Добавляем более легковесные WebP-версии, которые будут показаны пользователю по умолчанию, а если что-то пойдет не так браузер отобразит .png/.jpg аналоги.

Ориентиры для адекватных размеров изображений:

- декоративный hero-фон на всю ширину: примерно `1920-2560px` по ширине, обычно до `300-500 KB` в WebP;
- большие иллюстрации секций: экспортировать близко к реальному размеру в макете или до `2x`, но чаще держать в пределах `800-1400px` по большей стороне;
- изображения карточек участников: если в CSS они отображаются как `320x320px`, достаточно исходника `640x640px` для четкости на retina;
- небольшие иконки и стрелки лучше хранить в SVG, а не в PNG;
- если картинка после экспорта весит больше `300 KB`, ее стоит проверить и сжать; для hero-фона допустим чуть больший вес, потому что он крупный и заметный;
- не нужно оставлять огромные исходники вроде `3000-4000px`, если на странице они никогда не показываются в таком размере.

Чем сжимать изображения:

- вручную и наглядно: [Squoosh](https://squoosh.app/) от Google — удобно сравнивать качество до/после, менять формат, качество и размеры;
- через консоль для WebP: `cwebp -q 75 input.png -o output.webp`;
- для PNG: `pngquant --quality=65-80 --output output.png --force input.png`;
- для JPG: `jpegoptim --max=78 --strip-all input.jpg` или `mozjpeg`;
- на macOS можно использовать встроенный `sips`, но он обычно менее гибкий, чем специализированные инструменты;
- если нужно автоматизировать сжатие пачкой, можно написать небольшой Python-скрипт на Pillow и прогнать всю папку `assets/images/`.

Пример fallback для картинки:

```html
<picture>
  <source srcset="./assets/images/example.webp" type="image/webp" />
  <img src="./assets/images/example.png" alt="Описание изображения" />
</picture>
```

## 3. Подключить шрифты

Файлы шрифтов кладем в `assets/fonts/`. Для веба лучше использовать WOFF2 и понятные имена файлов без служебных суффиксов.

В этом проекте шрифты подключаются локально, а не запросом к Google Fonts. Выгода такая:
- страница не зависит от внешнего сервиса и не ломается, если Google Fonts недоступен;
- меньше внешних DNS/TLS-запросов при загрузке лендинга;
- проще контролировать конкретные начертания и веса, которые реально используются;
- локальные файлы можно кешировать вместе с остальной статикой проекта;
- это лучше для приватности: браузер пользователя не делает отдельный запрос на сторонний домен.

Подключение в `css/fonts.css`, импорт через `css/styles.css`. В проекте используются два семейства:
- `Merriweather` — для крупных заголовков;
- `Golos Text` — для основного текста, кнопок, таблиц и карточек.

## 4. Собрать семантический HTML-скелет

В `index.html` собираем смысловую структуру:

- `header` — логотип;
- `main` — основной контент;
- секции `hero`, `about`, `plan`, `participants`;
- повторяющаяся бегущая строка `ticker`;
- `footer` — дисклеймер.

У каждой крупной секции должен быть заголовок и связь через `aria-labelledby`. Если есть подзаголовок, который уточняет смысл секции, можно добавить `aria-describedby`.

Какие A11y-атрибуты и приемы здесь используются:

- `lang="ru"` на `html` — помогает скринридеру выбрать правильный язык произношения;
- `aria-labelledby` у секций — связывает блок с видимым заголовком, чтобы в навигации по landmarks было понятно, куда попал пользователь;
- `aria-describedby` у секции `plan` — добавляет к заголовку уточняющий подзаголовок;
- `aria-label` у каруселей, viewport и кнопок — дает понятное текстовое имя интерактивным зонам и кнопкам-стрелкам;
- `aria-roledescription="карусель"` — уточняет тип сложного виджета для скринридера;
- `aria-controls` у кнопок каруселей — показывает, каким viewport управляет кнопка;
- `aria-disabled` вместе с `disabled` — явно сообщает состояние недоступной кнопки;
- `aria-live="polite"` у счетчика карусели — позволяет озвучивать изменение номера слайда без резкого прерывания пользователя;
- `aria-hidden="true"` у декоративных повторов, точек и иконок — убирает визуальный декор из дерева доступности;
- `alt=""` у декоративных изображений — скринридер пропускает картинку, если смысл уже передан текстом рядом;
- `tabindex="0"` у viewport карусели — делает область фокусируемой и позволяет управлять ей с клавиатуры;
- класс `visually-hidden` — оставляет текст доступным для скринридеров, но визуально скрывает его, например для бегущей строки.

## 5. Добавить BEM-классы

После скелета добавим классы по BEM:

- блоки: `hero`, `about`, `plan`, `participants`, `carousel`, `ticker`, `footer`;
- элементы: `hero__title`, `about__details`, `plan__item`, `participant-card__image`;
- модификаторы: `button--primary`, `button--secondary`, `plan__text--wider`;
- утилиты: `u-relative`, `u-text-accent`, `u-section-padding`, `u-hidden-mobile`, `visually-hidden`.

Утилиты нужны для маленьких повторяемых правил, которые не принадлежат конкретному блоку: относительное позиционирование, акцентный цвет, общий вертикальный padding секций, скрытие элементов на мобильных или визуальное скрытие текста для скринридеров. Например, для акцентного фрагмента текста будем использовать утилиту вроде `u-text-accent`, а не модификатор без базового класса.

## 6. Связать HTML-скелет с CSS-классами и JS скриптами

После семантического скелета добавим к разметке BEM- и utility-классы, а затем последовательно наполним их стилями в `css/components.css`: общие элементы (`container`, заголовки, кнопки), сетки секций, фоновые слои, состояния интерактивных элементов, карусели и JS-хуки. HTML при этом отвечает за смысл и структуру, а CSS — за визуал, раскладку, адаптив и декоративные эффекты.

## 7. Настроить адаптив

Добавим media query для планшетов и мобильных: уменьшим заголовки, перестроем grid сетки, кнопки и карточки сделаем удобными для узкого экрана.
Пользователь может пользоваться лендингом на планшете и мобильном, но текущий адаптив не полностью соответствует дизайну и может быть доработан в будущем.

## 8. Проверить доступность

Минимальный чек-лист:

- у страницы есть корректный `lang="ru"`;
- все секции связаны с заголовками через `aria-labelledby`;
- декоративные картинки имеют `alt=""`;
- информативные картинки имеют осмысленный `alt`;
- кнопки каруселей имеют понятные `aria-label`;
- ссылки «Подробнее» уточняют, про какого участника идет речь;
- у бегущей строки есть текстовая версия для скринридеров;
- футеру достаточно семантического `<footer>`, если внутри только один дисклеймер;
- интерактивные элементы доступны с клавиатуры и имеют focus-состояния.

Как проверить верстку скринридером если у тебя macOS:

- на macOS включи VoiceOver: `Cmd + F5`;
- открой страницу в Safari или Chrome;
- пройди по странице клавишей `Tab` и проверь, что фокус попадает на кнопки, ссылки и viewport карусели в понятном порядке;
- используй `Control + Option + Right/Left`, чтобы последовательно слушать элементы страницы;
- открой rotor VoiceOver через `Control + Option + U` и проверь списки;
- после проверки выключи VoiceOver той же командой `Cmd + F5`.

## 9. Проверить совместимость

Для современных Chrome, Firefox, Edge и Safari дополнительных полифиллов обычно не нужно. 
Но в проекте используются возможности, которые чувствительны к старым браузерам:

- CSS Grid;
- `scroll-snap-type`;
- `image-set()`;
- `Element.scrollTo({ behavior: "smooth" })`;
- optional chaining в JS;
- `matchMedia().addEventListener("change", ...)`.

Если нужно поддерживать старые Safari, Android WebView или IE11, понадобится отдельная стратегия: Babel/транспиляция, fallback для `matchMedia.addListener`, упрощенная сетка или graceful degradation. Для IE11 полноценная поддержка такой верстки без серьезных упрощений нецелесообразна.

Дополнительно проверим страницу через Lighthouse в Chrome DevTools. Он покажет:

- `Performance` — насколько быстро грузятся шрифты, изображения, CSS и JS;
- `Accessibility` — базовые проблемы с alt, aria-атрибутами, контрастом, фокусом и именами кнопок;
- `Best Practices` — ошибки в консоли, небезопасные ссылки, устаревшие API;
- `SEO` — наличие `title`, `meta description`, корректной структуры ссылок и индексации.

Lighthouse не заменяет ручную проверку макета и скринридера, но хорошо подсвечивает технические проблемы, которые легко пропустить.

## 9. Проверить Pixel Perfect

Чтобы сверить верстку с макетом в режиме pixel-perfect, поставим расширение для наложения макета поверх страницы. Загружаем в него экспорт экрана из Figma, совмещаем ширину viewport с макетом, подстраеваем масштаб и прозрачность слоя и сравниваем ключевые блоки. На всех разрешениях идеал не гарантируется, но расхождения по отступам и размерам видно сразу.
